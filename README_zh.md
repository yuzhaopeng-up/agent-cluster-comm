> **多Agent集群5层通信架构** — 不是单一消息方案，而是完整的通信协议栈：加密P2P、消息总线、群聊协作、异步交接、健康监控。每层各有独特定位，组合起来覆盖从机密1:1到跨时区N:N协作的全频谱场景。

## 为什么需要5层？

没有单一的通信通道能覆盖所有多Agent场景：

| 场景 | 你需要的 | 单一通道做不到的 |
|------|---------|----------------|
| 发送含个人隐私的信用审查 | 端到端加密 | 消息总线会暴露给所有订阅者 |
| 向所有Agent广播"配置已更新" | 1:N实时推送 | P2P需要N次单独调用 |
| 人类实时观察Bot协商过程 | 可见、富媒体线程 | 消息总线对人类不可见 |
| UTC+8的Agent向UTC-5的Agent交接 | 异步、零部署、跨时区 | 消息总线要求双方同时在线 |
| 两个Agent报告矛盾的心跳 | 跨通道健康融合 | 只检查一个通道会漏掉另一个 |

**核心洞察**：你不需要选择一个通道——而是像网络协议栈一样分层。每层负责自己最擅长的场景，故障时逐层降级。

## 5层架构

```
┌─────────────────────────────────────────────────────────────┐
│  L5: 健康监控              cluster-health-monitor            │
│  监视所有层，触发故障切换                                      │
├─────────────────────────────────────────────────────────────┤
│  L4: 异步交接              github-async-handoff             │
│  跨时区、零部署、Issue即工单                                  │
├─────────────────────────────────────────────────────────────┤
│  L3: 群聊协作              group-chat-bot                   │
│  人类可见、富媒体、Bot间协商                                   │
├─────────────────────────────────────────────────────────────┤
│  L2: 消息总线              redis-message-bus                │
│  1:N广播、持久化队列、服务发现                                 │
├─────────────────────────────────────────────────────────────┤
│  L1: 加密P2P               encrypted-p2p-messaging          │
│  端到端加密1:1、穿越防火墙、webhook推送                        │
└─────────────────────────────────────────────────────────────┘
```

| 层 | 技能 | 拓扑 | 延迟 | 持久化 | 加密 | 人类可见 |
|----|------|------|------|--------|------|---------|
| L1 | encrypted-p2p-messaging | 1:1 | 中(中继) | Relay暂存 | 端到端 | 否 |
| L2 | redis-message-bus | 1:N / N:N | 低(亚毫秒) | Stream持久 | 传输层 | 否 |
| L3 | group-chat-bot | N:N | 中(API) | 平台存储 | 平台级 | **是** |
| L4 | github-async-handoff | 1:1 / 1:N | 小时级OK | Issue+Git历史 | HTTPS+ACL | **是** |
| L5 | cluster-health-monitor | 监视 | 检查周期 | 告警历史 | — | **是**(通过L3) |

## 场景选择指南

### 何时用L1 — 加密P2P

| 你应该在以下情况使用P2P... | 示例 |
|---|---|
| 消息包含PII、凭证或金融数据 | 信用审查、合规查询、KYC数据交换 |
| 需要特定Agent间的机密1:1通信 | 法律审查Bot向合规Bot发送特权文件 |
| Agent在不同防火墙/NAT后，无法直连 | 云端Agent与本地Agent通信 |

### 何时用L2 — 消息总线

| 你应该在以下情况使用消息总线... | 示例 |
|---|---|
| 需要向所有Agent广播通知 | "配置已更新"、"模型需重载"、"市场数据刷新" |
| 需要持久化队列和可靠投递 | 带审计跟踪的任务分发、断线重连后消息回放 |
| 需要服务发现——"哪些Agent能做X？" | 查找所有具有"异常检测"能力的Agent |

### 何时用L3 — 群聊协作

| 你应该在以下情况使用群聊... | 示例 |
|---|---|
| 人类需要观察协作过程 | 金融分析团队，人类分析师在环 |
| 需要富媒体输出（卡片、表格、图表） | 风险看板卡片、交互式审批按钮 |
| 需要人类实时介入 | "我应该执行这笔交易吗？" — 人类说是/否 |
| Bot需要在可见线程中协商 | Bot A提分析方案，Bot B建议替代方案，人类选择 |

### 何时用L4 — 异步交接

| 你应该在以下情况使用异步交接... | 示例 |
|---|---|
| Agent在不同时间/时区工作 | UTC+8团队收工，UTC-5团队明天开始 |
| 需要零部署协作（无Redis、无Relay） | 开源协作、黑客松、课堂教学 |
| 需要内置制品传输（代码、数据、文档） | CI/CD管线：构建→测试→部署交接链 |
| 需要完整审计跟踪以满足合规 | 受监管行业要求不可变、带时间戳的记录 |

### 何时用L5 — 健康监控

| 你应该在以下情况使用健康监控... | 示例 |
|---|---|
| 集群有3+个Agent需要保持协调 | 生产金融分析集群 |
| 需要检测和恢复部分故障 | 一个Agent变慢，其他Agent需要重新分配其任务 |
| 需要通信层之间的自动故障切换 | L2(Redis)宕机 → 自动切换到L4(GitHub) |

## 组合模式

### 模式1：实时分析集群（L1 + L2 + L3）

```
                  ┌───────────────────────────────┐
                  │   群聊 (L3)                     │
                  │   人类可见线程                   │
                  │   "风险分析启动中..."             │
                  └──────────┬────────────────────┘
                             │ 观察
┌──────────┐  L2: 广播       ┌──────────┐  L1: 加密       ┌──────────┐
│ Agent A   │ ──────────────> │ Agent B   │ <─────────────> │ Agent C   │
│ 协调者    │  "启动查询"      │ 执行者    │   信用审查       │ 合规审查  │
└──────────┘                  └──────────┘   (PII加密)       └──────────┘
```

**运作方式**：Agent A通过L2（消息总线）广播任务。Agent B接取并执行。对于包含PII的合规子任务，Agent B使用L1（加密P2P）与Agent C通信。全程L3（群聊）展示所有状态更新，人类可观察。

**典型场景**：金融分析、医疗诊断、法律审查。

### 模式2：跨时区研究团队（L2 + L3 + L4）

```
┌──────────┐                  ┌──────────┐
│ Agent A   │ ── L4: 交接 ───> │ Agent B   │
│ UTC+8     │  GitHub Issue    │ UTC-5     │
│ 晚上      │  "数据已收集"     │ 早上      │
└────┬─────┘                  └────┬─────┘
     │ L3: 状态更新                  │ L3: "继续分析"
     ▼                               ▼
┌──────────────────────────────────────────┐
│         群聊 (L3)                          │
│  "Agent A: 今日数据收集完成                  │
│   Agent B: 接续A的工作继续分析"              │
└──────────────────────────────────────────┘
```

**运作方式**：Agent A结束工作日，创建GitHub Issue（L4）并推送今日成果到分支。Agent B开始工作日，认领Issue，拉取数据，继续工作。双方都在群聊（L3）中发状态更新，团队看到连续性。

**典型场景**：分布式研究团队、开源协作、教育/课堂教学。

### 模式3：故障切换链（L2 → L4，由L5触发）

```
          L5检测到L2故障
                 │
                 ▼
    ┌─────────────────────────────┐
    │  正常: L2 (消息总线)         │ ──X── 宕机
    │  降级: L4 (异步交接)         │ ──→  激活
    └─────────────────────────────┘
         │                    │
         ▼                    ▼
    L3: 告警已发布        L4: 所有Agent
    "Redis不可达"         切换到Issue
```

**运作方式**：L5（健康监控）检测到Redis消息总线不可达。立即在群聊（L3）发布CRITICAL告警，并通过P2P（L1）通知。所有Agent切换到L4（GitHub异步交接）进行Agent间协调。Redis恢复后，Agent将未解决的Issue回放到消息总线。

**典型场景**：要求高可用的生产集群、通信故障会导致数据级联丢失的系统。

### 模式4：安全多方计算（L1 + L5）

```
┌──────────┐   L1: 加密         ┌──────────┐   L1: 加密         ┌──────────┐
│ Agent A   │ <────────────────> │ Agent B   │ <────────────────> │ Agent C   │
│有数据A    │  "请求B的数据"      │有数据B    │  "请求C的数据"      │有数据C    │
└──────────┘                     └──────────┘                     └──────────┘
     ▲                                ▲                                ▲
     │            L5: 健康监控         │                                │
     └─────────────────────────────────┴────────────────────────────────┘
```

**运作方式**：三个Agent各持有不同的敏感数据，需计算联合结果但不暴露各自的输入。所有通信使用L1（加密P2P）——不经消息总线、不经群聊。L5监控三个Agent全程在线和响应。

**典型场景**：多方风险评分、跨机构合规验证、隐私保护分析。

## 故障切换决策流

```
Agent需要发送消息
         │
         ▼
   含PII/敏感数据？
    ├── 是 → 使用L1（加密P2P）
    │         │
    │         ▼
    │    接收方在线？
    │     ├── 是 → 通过L1发送
    │     └── 否 → 排入L4（异步交接），通过L3通知
    │
    └── 否 → 需要广播(1:N)？
             ├── 是 → L2(消息总线)可用？
             │        ├── 是 → 使用L2
             │        └── 否 → 降级到L4(为每个接收方创建Issue)
             │                  + 通过L3和L5告警通知
             │
             └── 否 → 需要人类可见？
                      ├── 是 → 使用L3（群聊）
                      └── 否 → 根据延迟需求选L1(P2P)或L2(总线)
```

## 快速开始

```bash
# 克隆仓库
git clone https://github.com/yuzhaopeng-up/agent-cluster-comm.git

# 安装5个技能
cp -r agent-cluster-comm/skills/* ~/.config/TeleAgent/skills/
```

### 最小可用集群（2层）

从简开始，只需L1 + L2：

```bash
# L1: 加密P2P用于敏感数据
pip install encrypted-p2p-messaging

# L2: Redis消息总线用于广播
docker run -d --name redis-bus -p 6379:6379 redis:7-alpine
pip install redis-message-bus
```

### 完整5层栈

```bash
# L1: 加密P2P
docker run -d --name p2p-relay -p 8765:8765 your-org/p2p-relay:latest

# L2: Redis消息总线
docker run -d --name redis-bus -p 6379:6379 redis:7-alpine

# L3: 群聊Bot（配置你的平台凭证）
export CHAT_PLATFORM=feishu
export CHAT_APP_ID=your-app-id
export CHAT_APP_SECRET=your-app-secret

# L4: GitHub异步交接（零部署）
export GITHUB_TOKEN=ghp_xxxxxxxxxxxx
export HANDOFF_REPO=your-org/agent-workspace

# L5: 集群健康监控
cluster-health-monitor watch --interval 60
```

## 行业场景

| 行业 | L1 加密P2P | L2 消息总线 | L3 群聊协作 | L4 异步交接 | L5 健康监控 |
|------|-----------|-----------|-----------|-----------|-----------|
| 金融服务 | 信用审查机密交换 | 市场事件实时广播 | 分析师观察Bot协商 | 隔夜批处理交接 | 检测风控Agent延迟 |
| 医疗健康 | 患者数据加密传输 | 告警广播 | 医生观察AI诊断 | 实验室结果隔日审查 | 监控处理患者数据的Agent |
| DevOps | 凭证安全交换 | 事件告警广播 | 值班工程师观察自动响应 | 构建→测试→部署交接链 | 检测Agent故障并自动修复 |
| 教育科研 | 学生-教师私聊 | 作业广播 | 课堂可见协作解题 | 跨时区同行评审 | 监控学生Agent参与度 |
